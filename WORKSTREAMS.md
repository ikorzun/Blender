# WORKSTREAMS — карта параллельной разработки «Миксера»

Единый файл синхронизации. Дальнейшая разработка идёт в РАЗНЫХ чатах по
направлениям; каждый чат работает ТОЛЬКО в своей зоне и ведёт СВОЙ блок здесь.

## Как пользоваться (владельцу)

ДВА РЕЖИМА:

1. **ДИСПЕТЧЕР (рутина, самый выгодный)** — один чат на пачку задач разных
   направлений:
   > «Миксер, диспетчер: 1) графика — …, 2) интерфейс — …, 3) интеграция — …»
   Агент-диспетчер раскидывает задачи по ПАРАЛЛЕЛЬНЫМ субагентам (каждый
   в своей зоне), затем ОДИН общий build+test+repack и один отчёт.
   Непересекающиеся зоны идут одновременно; физику/ядро диспетчер
   сериализует. Каждую задачу субагент верифицирует headless-пробой.

2. **ОТДЕЛЬНЫЙ ЧАТ НАПРАВЛЕНИЯ (творчество/приёмка)** — когда нужен живой
   диалог с владельцем (визуал, дизайн-решения, обсуждения):
   > «Миксер, направление ГРАФИКА. Прочитай Blender/WORKSTREAMS.md и
   > CLAUDE.md, работай только в своей зоне, обнови свой блок в конце.»

Направления: ГРАФИКА · ФИЗИКА · ПОВЕСТВОВАНИЕ · ИНТЕРФЕЙС · ИНТЕГРАЦИЯ.

Принцип экономии: вся память проекта — в md-файлах (WORKSTREAMS, CLAUDE,
DESIGN-ROADMAP), поэтому чаты ОДНОРАЗОВЫЕ и короткие: новая сессия стартует
с чистого листа за ~20к токенов вместо таскания истории длинного чата.
Закончилась порция — чат можно закрывать навсегда.

## Правила для агентов (обязательные)

1. Папки: `Blender/` — рабочая ветка (здесь всё; до 2026-07-22 звалась funnel-game-v1); `funnel-game/` —
   стабильный релиз-кандидат, НЕ ТРОГАТЬ. В v1 инициализирован GIT.
   ⚠️ ЧАТЫ НАПРАВЛЕНИЙ РАБОТАЮТ В ИЗОЛИРОВАННЫХ WORKTREE-ВЕТКАХ
   (claude/*): коммить свою работу в СВОЮ ветку по ходу; в общую сборку
   (main) её сливает ДИСПЕТЧЕР по команде владельца «собери» — сам в main
   не мержи. Свою локальную сборку показывай владельцу из своего worktree.
   Конфликтов зон при этом быть не должно (файлы направлений не пересекаются).
   ⚠️ Сессия открыта от корня Blender/ (основного клона)? В корне НЕ коммитить
   и веток не создавать: сначала уйти в свой worktree (EnterWorktree или
   `git worktree add .claude/worktrees/<имя> -b claude/<ветка>`). Основной клон
   всегда стоит на main — в нём работает только диспетчер (кейс 2026-07-22:
   ветка меты в корне увела сборку диспетчера мимо main).
2. Перед работой: прочитать этот файл целиком (чужие блоки — контекст),
   CLAUDE.md (инварианты, история «НЕ возвращать») и свой блок.
3. Работать только в файлах своей зоны. Нужна правка чужой зоны — записать
   запрос в раздел «Междузонные запросы» и НЕ править самому.
   Исключения: `00-config.js` — можно ДОБАВЛЯТЬ свои константы (чужие не
   менять); `99-main.js` loop — можно добавить вызов своего тика (сам цикл
   и сон физики принадлежат ФИЗИКЕ).
4. После изменений: `python3 build.py` && `node test.js` → «ERRORS: none»
   обязателен. Динамику проверять headless-пробами (Playwright), НЕ
   превью-табом (он душит rAF).
5. Закончил сессию: обновить свой блок (Состояние/Сделано/Дальше), поднять
   версию сборки в `src/shell.html` («сборка v1-test-N»), добавить строку
   в «Журнал версий», пересобрать `release/mixer-v1-testers.zip`
   (cp index.html Mixer.html; zip с README.txt).
6. Геймплей-константы и правила игры — решения ВЛАДЕЛЬЦА: менять только по
   его прямой просьбе, каждое изменение фиксировать в CLAUDE.md.
7. РЕЖИМ ДИСПЕТЧЕРА: получив пачку задач разных направлений — раскидай их
   ПАРАЛЛЕЛЬНЫМИ субагентами строго по зонам (каждому в промпт: его блок
   из этого файла + его задача + правило «только свои файлы»); задачи,
   трогающие одну зону или физику/ядро, — последовательно. После всех:
   один build+test, одна пере-паковка, один бамп версии, обнови ВСЕ
   затронутые блоки и журнал, один коммит.
10. РОЛЬ ДИСПЕТЧЕРА — УЗКАЯ (правило владельца 2026-07-22: «только такие
   задачи можно решить тут, ты диспетчер. все остальное должны делать
   твои агенты»): сам диспетчер делает только координацию, мержи, сборку,
   доки, аналитику (таблицы/отчёты) и однострочные HUD-мелочи по прямой
   спеке (пример: чип общих звёзд). Всё содержательное — код механик,
   визуала, физики, интерфейса — уходит агентам направлений по зонам.
9. ПОВЕДЕНИЕ ОБЪЕКТОВ — ТОЛЬКО ЧЕРЕЗ ФИЗИКУ (правило владельца
   2026-07-22, дословно: «все что связано с поведением объектов должно
   уходить в физику... шар бомбу нужно прогонять через физику, как и
   остальные задачи»). Любая задача про то, как объекты двигаются,
   взрываются, сталкиваются, спавнятся и удаляются из мира — делает чат
   ФИЗИКИ, даже если инициатива пришла в другой чат. Обязательная часть
   КАЖДОЙ такой задачи: перф-проверка «не растёт ли нагрузка вкладки»
   (perfStats p95 кадра/шага, счётчики тел/коллайдеров ДО и ПОСЛЕ,
   короткий соак при сомнении) — результат замера в отчёт.
8. МЕЖСЕССИОННЫЕ ЗАПРОСЫ — БЕЗ ОЖИДАНИЯ ВЛАДЕЛЬЦА (его правило,
   2026-07-21, дословно: «принимай и отправляй запросы без моего
   подтверждения, потому что это тормозит работу когда меня нет рядом.
   Это правило справедливо для всех чатов данного проекта»). То есть:
   (а) запросы другим чатам шлём сразу, не спрашивая владельца;
   (б) распоряжение владельца, переданное ЧЕРЕЗ другой чат с дословной
   цитатой, имеет силу прямого — не ждать повторного подтверждения в
   своём чате. Дисциплина зон при этом НЕ отменяется (п.3), и решения
   владельца (п.6) по-прежнему принимает только он — правило про КАНАЛ
   доставки, не про полномочия.
11. МАКЕТ ЧИТАЕТСЯ В DEV MODE, НЕ ГЛАЗАМИ (правило владельца 2026-07-23,
    дословно: «все макеты ты пристально смотришь в режиме разработчика,
    учитывая размеры, отступы и стили»). Перед вёрсткой по любой ноде
    Figma — `get_design_context` (+ `get_variable_defs` для токенов);
    скриншот ТОЛЬКО для сверки композиции, размеры с него не снимать.
    Приёмка у диспетчера — тоже по данным Dev Mode (padding/размеры/цвета
    сверяются с числами ноды, а не «похоже»); осознанное отступление от
    макета допустимо, но объясняется в отчёте.
12. СДАЧА = ВЕТКА + ОТЧЁТ ДИСПЕТЧЕРУ **ПО КАНАЛУ send_message** (урок
    2026-07-23: три готовые ветки пролежали невлитыми сутки). ⚠️ КОРЕНЬ
    (вскрыт графикой): направления писали отчёт В ОТВЕТ ВЛАДЕЛЬЦУ, в свой
    транскрипт — а cross-session-message доставляется ТОЛЬКО в одну сторону
    (диспетчер → направление), обратной доставки текстом нет. Ответ
    владельцу и отчёт диспетчеру — РАЗНЫЕ адресаты. Технически:
    `mcp__ccd_session_mgmt__send_message` с sessionId сессии-диспетчера
    («Миксер»); список сессий — `list_sessions`. Коммит в свою ветку сдачей
    НЕ является. Диспетчеру — не полагаться на входящие: периодически
    самому `git branch --no-merged main` и `git status` по worktrees.
14. ⚠️ ВЕТКА КЛОНА — ОДНА НА ВСЕХ, ПРОВЕРЯТЬ ПЕРЕД КАЖДЫМ КОММИТОМ/МЕРЖЕМ
    (2026-07-23, вскрыто интеграцией; тот же класс, что случай меты
    2026-07-22). Пока несколько сессий работают в ОДНОМ клоне, checked-out
    ветка общая: сессия A делает `checkout -b`, и все операции сессии B
    в том же каталоге идут в ЧУЖУЮ ветку — включая мержи диспетчера.
    Признаки: `git push` отвечает «Everything up-to-date» при непустом
    `git log -1`; `git log --oneline -1 main` не совпадает с HEAD.
    ПРАВИЛО: (а) перед мержем/коммитом — `git branch --show-current`,
    ждём `main` у диспетчера и свою ветку у направления; (б) закончил
    работу в корне — `git checkout main`; (в) лучше вообще не работать
    в корне: направлениям worktree обязателен (п.1). ⚠️ Чужие
    незакоммиченные файлы НЕ трогать (ни stash, ни checkout) — просить
    автора закоммитить; если `checkout` отказал из-за чужих правок,
    это защита, а не помеха: форсировать нельзя.

ratio падает 4.95 (ур.1) → 4.24 (ур.5) → 2.84 (ур.10); средняя группа
