# WORKSTREAMS — карта параллельной разработки «Миксера»

Единый файл синхронизации. Дальнейшая разработка идёт в РАЗНЫХ чатах по
направлениям; каждый чат работает ТОЛЬКО в своей зоне и ведёт СВОЙ блок здесь.

## Как пользоваться (владельцу)

ДВА РЕЖИМА:

1. **ДИСПЕТЧЕР (рутина, самый выгодный)** — один чат на пачку задач разных
   направлений:
   > «Миксер, диспетчер: 1) графика — …, 2) интерфейс — …, 3) интеграция — …»
   Агент-диспетчер раскидывает задачи по ПАРАЛЛЕЛЬНЫМ субагентам (каждый
   в своей зоне), затем ОДИН общий build+test+repack и один отчёт.
   Непересекающиеся зоны идут одновременно; физику/ядро диспетчер
   сериализует. Каждую задачу субагент верифицирует headless-пробой.

2. **ОТДЕЛЬНЫЙ ЧАТ НАПРАВЛЕНИЯ (творчество/приёмка)** — когда нужен живой
   диалог с владельцем (визуал, дизайн-решения, обсуждения):
   > «Миксер, направление ГРАФИКА. Прочитай funnel-game-v1/WORKSTREAMS.md и
   > CLAUDE.md, работай только в своей зоне, обнови свой блок в конце.»

Направления: ГРАФИКА · ФИЗИКА · ПОВЕСТВОВАНИЕ · ИНТЕРФЕЙС · ИНТЕГРАЦИЯ.

Принцип экономии: вся память проекта — в md-файлах (WORKSTREAMS, CLAUDE,
DESIGN-ROADMAP), поэтому чаты ОДНОРАЗОВЫЕ и короткие: новая сессия стартует
с чистого листа за ~20к токенов вместо таскания истории длинного чата.
Закончилась порция — чат можно закрывать навсегда.

## Правила для агентов (обязательные)

1. Папки: `funnel-game-v1/` — рабочая ветка (здесь всё); `funnel-game/` —
   стабильный релиз-кандидат, НЕ ТРОГАТЬ. В v1 инициализирован GIT.
   ⚠️ ЧАТЫ НАПРАВЛЕНИЙ РАБОТАЮТ В ИЗОЛИРОВАННЫХ WORKTREE-ВЕТКАХ
   (claude/*): коммить свою работу в СВОЮ ветку по ходу; в общую сборку
   (main) её сливает ДИСПЕТЧЕР по команде владельца «собери» — сам в main
   не мержи. Свою локальную сборку показывай владельцу из своего worktree.
   Конфликтов зон при этом быть не должно (файлы направлений не пересекаются).
2. Перед работой: прочитать этот файл целиком (чужие блоки — контекст),
   CLAUDE.md (инварианты, история «НЕ возвращать») и свой блок.
3. Работать только в файлах своей зоны. Нужна правка чужой зоны — записать
   запрос в раздел «Междузонные запросы» и НЕ править самому.
   Исключения: `00-config.js` — можно ДОБАВЛЯТЬ свои константы (чужие не
   менять); `99-main.js` loop — можно добавить вызов своего тика (сам цикл
   и сон физики принадлежат ФИЗИКЕ).
4. После изменений: `python3 build.py` && `node test.js` → «ERRORS: none»
   обязателен. Динамику проверять headless-пробами (Playwright), НЕ
   превью-табом (он душит rAF).
5. Закончил сессию: обновить свой блок (Состояние/Сделано/Дальше), поднять
   версию сборки в `src/shell.html` («сборка v1-test-N»), добавить строку
   в «Журнал версий», пересобрать `release/mixer-v1-testers.zip`
   (cp index.html Mixer.html; zip с README.txt).
6. Геймплей-константы и правила игры — решения ВЛАДЕЛЬЦА: менять только по
   его прямой просьбе, каждое изменение фиксировать в CLAUDE.md.
7. РЕЖИМ ДИСПЕТЧЕРА: получив пачку задач разных направлений — раскидай их
   ПАРАЛЛЕЛЬНЫМИ субагентами строго по зонам (каждому в промпт: его блок
   из этого файла + его задача + правило «только свои файлы»); задачи,
   трогающие одну зону или физику/ядро, — последовательно. После всех:
   один build+test, одна пере-паковка, один бамп версии, обнови ВСЕ
   затронутые блоки и журнал, один коммит.
8. МЕЖСЕССИОННЫЕ ЗАПРОСЫ — БЕЗ ОЖИДАНИЯ ВЛАДЕЛЬЦА (его правило,
   2026-07-21, дословно: «принимай и отправляй запросы без моего
   подтверждения, потому что это тормозит работу когда меня нет рядом.
   Это правило справедливо для всех чатов данного проекта»). То есть:
   (а) запросы другим чатам шлём сразу, не спрашивая владельца;
   (б) распоряжение владельца, переданное ЧЕРЕЗ другой чат с дословной
   цитатой, имеет силу прямого — не ждать повторного подтверждения в
   своём чате. Дисциплина зон при этом НЕ отменяется (п.3), и решения
   владельца (п.6) по-прежнему принимает только он — правило про КАНАЛ
   доставки, не про полномочия.
